Weekend Projects

by Brendan Horng | 2017-06-21

Dotfiles

I set up a dotfile directory to house all of my dotfiles this past weekend to automatically assist in setting up my dotfiles and syncing them.

Motivation

I decided to do this as I am moving towards using the command line and Unix to do most of my development. This involves Tmux and Vim, which I have been using for a while now.

The Project

The project basically consists of my dotfiles dumped into a directory. This is currently just .vimrc and .tmux.conf, though I expect that this list will grow a little as time goes on.

I also wrote two custom shell scripts to help with installation and syncing.

File Locations

This is just a file of the format FILE_NAME FILE_PATH per line. FILE_NAME would be the name of the dotfile and FILE_PATH would be the path that it is expected to be at in my system. All paths are relative to the $HOME directory (in the case of my desktop, /home/brendan).

An example of this would be: .vimrc.

Because I specify no path, .vimrc will simply be expected to be in $HOME.

Setup

The setup script reads from file_locations and runs through and checks if the file exists at its expected location. Then:

  • If the files are different:
    • The file gets copied over to the destination
  • If the files are the same:
    • The file gets copied over if the file is different than the one in the repository
Sync

The sync_dotfiles script checks if the repository is currently on master. If it isn't, it stashes local changes and switches to master. It then checks if we currently have the latest version of master:

  • If we are not on the latest version of master:
    1. Pulls in the latest changes of master
    2. Overwrites each file locally to match master based on file_locations
  • If we are on the latest version of master:
    1. Goes through file_locations and copies over any changes into the repo
    2. Commits and pushes up an automated backup to master if any of the scripts have changed

The sync_dotfiles script is treated as a "dotfile" itself. Meaning it can overwrite/backup itself and is included in the setup.

Now I know what you are thinking: "That sounds awfully dangerous."

And it is, but it is also awfully convenient. My argument is that I don't need to make this entirely idiot proof as I probably will be the only one using this (not that I'm not an idiot, but I think I can trust myself to not screw up this system).

Another hole in this script is that it could potentially overwrite changes locally on one device if it is behind on master. This is true. But again, I don't expect that I will be changing my dotfiles ALL that often, and rarely on two different computers at once.

So yes, it could cause problems, but none that are significant enough for me to feel like adding extra complexity to this system. It will work fine for what I need.

This sync_dotfiles script is added to my /etc/cron.daily so it will perform the backup automatically everyday.

Bigger Picture

This is eventually going to be part of a bigger system. I am going to make scripts to automate the setup of my Linux work environment to expedite the process when I switch Linux distros or decide to wipe my partition (fastest way to get of junk is to start fresh in my opinion).

This is a good start for me and I am excited to see how this works now that I have begun using it.
